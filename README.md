# VideoHub — видеохостинг с ML-рекомендациями и модерацией

Полноценный видеохостинг уровня «мини-YouTube»: загрузка и просмотр видео,
каналы, подписки, комментарии, монетизация (пре-ролл реклама, премиум-подписка,
выплаты авторам), админ-панель — и поверх всего этого **ML-движок рекомендаций**
и **система Trust & Safety** (жалобы + автоматическая модерация контента).

* **Frontend:** React 18 + TypeScript + Vite + TailwindCSS + Zustand + TanStack Query.
* **Backend:** Go 1.22 (chi, pgx/v5, JWT HS256, bcrypt).
* **ML/аналитика:** собственный гибридный рекомендатель + лексико-эвристический
  классификатор модерации (пакет [`backend/internal/ml`](backend/internal/ml)).
* **БД:** PostgreSQL 16 — схема и seed применяются автоматически при старте.
* **Безопасность:** выделенный слой ([`backend/internal/security`](backend/internal/security)) —
  security-заголовки, rate limiting, защита auth-эндпоинтов.
* **Оркестрация процессов:** BPMN/DMN-модели (Camunda 7/8) для модерации,
  выплат, онбординга партнёров и DMCA — см. раздел ниже.
* **Контейнеризация:** docker compose — весь стек одной командой.

## Насколько это сложный проект

Это не CRUD-«тудушка». Ключевая инженерная сложность распределена по нескольким слоям:

| Слой | Что нетривиального |
|------|--------------------|
| **Рекомендации** | Гибрид из 5 методов (популярность, качество, контентная близость, коллаборативная фильтрация, персонализация) + пере-ранжирование MMR ради разнообразия. Не одна формула `ORDER BY`, а полноценный ranking-pipeline. |
| **Модерация** | ML-классификатор с покомпонентными скорами (nudity/copyright/spam/violence), пороговой логикой решений и **DMN-матрицей санкций**. Каждое решение объяснимо (labels) и логируется (audit trail). |
| **Trust & Safety** | Пользовательские жалобы с приоритизацией очереди, статусами и анти-флудом (один открытый репорт на объект от пользователя — на уровне уникального индекса БД). |
| **Монетизация** | Денормализованные счётчики + event-таблицы в одной транзакции; баланс канала, транзакции, выплаты, премиум. |
| **Данные** | Согласованность денормализованных счётчиков через транзакции; админ-оверрайды, применяемые на лету в SQL; аналитика по дням из event-таблиц. |
| **Безопасность** | Отдельный middleware-слой: CSP/HSTS/anti-clickjacking, token-bucket rate limiter на IP, отдельный жёсткий лимит на brute-force авторизации. |
| **Оркестрация** | Бизнес-процессы (модерация, выплаты, онбординг, DMCA) вынесены из кода в BPMN/DMN — меняются без релиза. |

## Архитектура

```
Браузер (SPA, React)
        │  HTTPS
        ▼
   nginx  ──►  /api, /uploads → Go backend (chi)
        │
        ▼ (статика SPA)
                     ┌──────────────────────────────────────────────┐
                     │              Go backend (chi)                 │
                     │                                               │
   middleware ──►    │  SecureHeaders → RateLimiter → CORS → JWT     │
                     │                                               │
   handlers  ──►     │  videos / auth / channels / comments /        │
                     │  reports / moderation / finance / admin       │
                     │        │                    │                 │
   бизнес    ──►     │   internal/ml         internal/security       │
                     │   (recommend +         (headers, rate limit)  │
                     │    moderation)                                │
                     │        │                                      │
   store     ──►     │   internal/store (pgxpool, транзакции)        │
                     └──────────────────────────────────────────────┘
                                        │
                                        ▼
                                 PostgreSQL 16
```

Принципы:
- **Тонкие хендлеры, толстый store.** HTTP-слой только валидирует и делегирует; вся работа с данными и целостность — в `store` (транзакции pgx).
- **Денормализация + event-таблицы.** Счётчики (`views_count`, `likes_count`…) живут рядом с сущностью и инкрементируются в одной транзакции с записью в журнал событий — это даёт и быстрые чтения, и аналитику по дням.
- **ML — чистый пакет без БД.** [`internal/ml`](backend/internal/ml) не знает про PostgreSQL: получает данные на вход, возвращает ранжирование/вердикт. Легко тестировать и заменять модели.
- **Безопасность как слой, а не как if-ы в хендлерах.** Вся политика — в [`internal/security`](backend/internal/security).

Подробная карта пакетов — в [backend/CLAUDE.md](backend/CLAUDE.md) и [src/CLAUDE.md](src/CLAUDE.md).

## 🤖 ML-движок рекомендаций

Код: [`backend/internal/ml/recommend.go`](backend/internal/ml/recommend.go),
[`features.go`](backend/internal/ml/features.go).

Рекомендации строятся **не «как попало»**, а как взвешенная комбинация пяти
независимых методов, каждый закрывает свой изъян остальных:

| # | Метод | Что делает | Защищает от |
|---|-------|-----------|-------------|
| 1 | **Popularity** | Вовлечённость (`views + likes*5 − dislikes*3`) с log-сжатием + экспоненциальный бонус свежести (half-life 7 дней). | Застоя ленты, «мёртвых» видео. |
| 2 | **Quality (Wilson)** | Нижняя граница 95% доверительного интервала доли лайков. | Обмана малой выборкой: «3 лайка / 0 дизлайков» не побеждает «900 / 100». |
| 3 | **Content-based** | Косинусная близость векторов признаков (категория, теги, длительность, токены заголовка) к затравке. | Нерелевантных «похожих» рядом с плеером. |
| 4 | **Collaborative** | Item-item ко-просмотры: «те, кто смотрел это, смотрели также…» (считается из `video_views` в SQL). | Замыкания только на метаданных — ловит неочевидные связи. |
| 5 | **Personalized** | Аффинность зрителя к категориям (из его истории просмотров) + буст видео каналов, на которые он подписан. | Обезличенной выдачи одинаковой для всех. |

**Гибрид → разнообразие.** Итоговый скор — линейная комбинация с разными весами
для главной ленты (упор на персонализацию) и для блока «похожие» (упор на
контент + коллаборацию). Затем список пере-ранжируется алгоритмом
**MMR (Maximal Marginal Relevance)**: на каждом шаге выбирается кандидат с лучшим
балансом *релевантности* и *непохожести на уже выбранное* — чтобы лента не
забивалась одним каналом или одной темой. Уже просмотренные видео сильно
понижаются, но не исчезают совсем.

```
итоговый_балл(v) = w1·popularity + w2·quality + w3·content + w4·collab + w5·personal
лента           = MMR( кандидаты, λ=0.72 )   // 0.72 релевантность / 0.28 разнообразие
```

Для анонимного зрителя персонализация и коллаборация вырождаются в 0 — движок
корректно деградирует до «популярное + качественное + свежее».

Эндпоинты: `GET /api/videos/recommended` (главная, персонализированная),
`GET /api/videos/{id}/recommendations` (похожие рядом с плеером).

## 🛡️ Trust & Safety: модерация и жалобы

### Автоматическая ML-модерация

Код: [`backend/internal/ml/moderation.go`](backend/internal/ml/moderation.go).

Каждое загруженное видео **до попадания в ленты** прогоняется через классификатор.
Он оценивает риск по четырём меткам и агрегирует их (принцип «сильнейшее
нарушение решает» + затухающий вклад остальных):

| Метка | Как оценивается |
|-------|-----------------|
| `nudity` | лексикон adult-контента по заголовку/описанию/тегам |
| `violence` | лексикон жестокости/терроризма |
| `copyright` | лексикон правообладателей + эвристика «полный фильм/эпизод» на длинных роликах |
| `spam` | капс, повторяющиеся символы, число ссылок, «раздачи призов», теговый спам, крипто-паттерны |

Скоры сжимаются логистикой в вероятности, из них считается общий риск, и по
**порогам** выносится решение:

```
overall ≥ 0.80  → auto_block      (moderation_status = blocked)   — скрыто сразу
0.40 ≤ o < 0.80 → manual_review    (moderation_status = pending)   — в очередь модератору
overall < 0.40  → approved         (moderation_status = approved)  — публикуется
```

> Пороги специально совпадают с BPMN-моделью процесса модерации (см. ниже) —
> ML-часть и оркестрация говорят на одном языке.

**DMN-матрица санкций.** Вместо `switch` из десятков кейсов — таблица правил
«тип нарушения → санкция» (`warning` / `demonetize` / `hide` / `ban`). Итоговая
санкция — самая строгая из сработавших. За spam/nudity/violence возможен бан
автора, за copyright — только скрытие/страйк.

Каждый прогон объясним (`labels` — что именно сработало) и сохраняется в
`moderation_results` как **audit trail**. Модератор видит очередь с покомпонентными
скорами и может: одобрить / заблокировать / выдать теневой бан / перепрогнать ML
после правки метаданных (`/admin/moderation`).

Статус `moderation_status` фильтрует выдачу: в публичные ленты и рекомендации
попадают только `approved`.

### Жалобы пользователей

Код: [`store/reports.go`](backend/internal/store/reports.go),
[`handlers/reports.go`](backend/internal/handlers/reports.go),
UI: кнопка «Пожаловаться» на странице видео + вкладка `/admin/reports`.

- Любой залогиненный пользователь жалуется на видео/комментарий/канал с указанием
  причины (`spam`, `nudity`, `violence`, `copyright`, `hate`, `misinformation`, `other`).
- **Приоритизация очереди**: причина (nudity/violence — выше) + тип объекта
  (публичное видео важнее комментария) → числовой приоритет; очередь сортируется
  по нему.
- **Анти-флуд** на уровне БД: уникальный частичный индекс не даёт одному
  пользователю держать больше одной открытой жалобы на один объект.
- Админ видит жалобы с фильтром по статусу, берёт в работу, подтверждает или
  отклоняет — с фиксацией, кто и когда закрыл (audit).

## 🔒 Безопасность

Отдельный слой [`backend/internal/security`](backend/internal/security), подключённый
глобально в роутере — политика собрана в одном месте, а не размазана по хендлерам.

**Security-заголовки** (`SecureHeaders`):
- `Content-Security-Policy` — `default-src 'self'`, медиа/картинки только со своего
  origin + `data:`/`blob:`, `object-src 'none'`, `frame-ancestors 'self'`.
- `X-Content-Type-Options: nosniff` — против MIME-sniffing.
- `X-Frame-Options: SAMEORIGIN` + CSP `frame-ancestors` — против clickjacking.
- `Referrer-Policy: strict-origin-when-cross-origin`, `Permissions-Policy` (гео/микрофон/камера off).
- `Strict-Transport-Security` — включается автоматически под HTTPS.

**Rate limiting** (token-bucket на IP, с фоновой уборкой бакетов):
- Глобальный щедрый лимит (30 rps, burst 60) — отсекает флуд/скрейпинг.
- **Жёсткий лимит на `/api/auth/*`** (≈1 запрос / 2 с) — защита от перебора
  паролей и кодов подтверждения.

**Прочее уже в проекте:**
- Пароли — bcrypt; JWT HS256 с TTL; гварды `RequireAuth` / `RequireAdmin`.
- Загрузки — проверка расширения и размера, файлы под UUID-именами.
- SQL — только параметризованные запросы (pgx), никаких конкатенаций.
- Уникальные просмотры берут id строго из JWT (не из query) — счётчик не накрутить анонимом.
- `DisallowUnknownFields` при разборе JSON-тел.

### Чек-лист перед продакшеном
1. `JWT_SECRET` — случайные ~64 байта.
2. `AUTH_EXPOSE_CODE=false` + реальная отправка email.
3. Сменить/удалить учётку `admin/admin1` из seed.
4. Сузить `CORS_ORIGINS` до своего домена.
5. Терминировать TLS на nginx/ingress (HSTS включится сам).

## ⚙️ Оркестрация бизнес-процессов (BPMN / DMN, Camunda 7 & 8)

Долгоживущие, юридически чувствительные и часто меняющиеся процессы вынесены из
кода сервисов в исполняемые BPMN-модели. Бизнес-логика остаётся в микросервисах
(external task workers на Python забирают задачи и возвращают результат), а
**оркестрация и правила** — в BPM-движке, где их правит product/юрист без релиза.

### 1. Процесс модерации контента (Camunda 7)
Раньше правила модерации были захардкожены в коде — менять их значило релизить.
Переведено на BPMN.

- **Модель:** загрузка видео → ML-проверка (nudity / copyright / spam) → если
  `score > 0.8` — автоблок; если `0.4–0.8` — ручная модерация; если `< 0.4` —
  публикация. Пороги совпадают с классификатором в `internal/ml`.
- **Параллельная ветка:** проверка аудио на копирайт через ACRCloud
  (audio fingerprinting).
- **External task workers на Python** забирают задачи из Camunda и возвращают
  результат — бизнес-логика в микросервисах, оркестрация в BPM.
- **User tasks для модераторов** с приоритизацией: жалобы на прямые эфиры → в
  начало очереди, обычные видео → FIFO.
- **DMN-таблица** для матрицы «тип нарушения → санкция» (предупреждение /
  демонетизация / скрытие / бан). Раньше — `switch` из ~40 кейсов, теперь таблица,
  которую редактирует product-менеджер без разработчика.

### 2. Процесс выплат авторам (Camunda 8 / Zeebe)
Ежемесячные выплаты по монетизации. Раньше — крон-скрипт на 800 строк, где нельзя
было понять, где что зависло.

- **Модель:** расчёт баланса → проверка порога выплаты ($100) → верификация
  налоговой формы (W-8/W-9 для нерезидентов) → выбор платёжного метода
  (PayPal / банковский перевод / ЮMoney) → отправка → ожидание подтверждения →
  уведомление автора.
- **Message events** — ожидание подтверждения от платёжной системы (до 3 дней):
  процесс висит в ожидании без блокировки ресурсов.
- **Compensation events** — если платёж провалился после списания с баланса
  компании, автоматически откатывается баланс автора и создаётся задача в саппорт.
- **Timer boundary events** — если платёж не подтверждён за 7 дней, эскалация на
  финансового менеджера.

### 3. Онбординг стримеров-партнёров
Процесс с человеческими шагами: заявка → проверка канала (≥1000 подписчиков,
≥4000 часов просмотра) → подписание оферты через DocuSign → настройка монетизации
→ welcome-email.

- Смоделировано в Camunda Modeler; интеграция с **DocuSign** через service task
  (REST-коннектор).
- **Call activity** для переиспользования подпроцесса **KYC-верификации** — тот же
  подпроцесс используется в выплатах и при регистрации донат-плательщиков.

### 4. Обработка жалоб DMCA
Юридически чувствительный процесс, где важна прослеживаемость каждого шага.

- **Полный цикл DMCA:** получение жалобы → уведомление автора → 10 дней на
  counter-notice → блокировка или разблокировка → уведомление заявителя.
- Все шаги логируются в **Camunda History** — полный audit trail для юристов
  (раньше приходилось собирать по логам разных сервисов).

> В этом репозитории реализованы прикладные сервисы (ML-модерация, выплаты,
> баланс, жалобы) и их API; BPMN/DMN-слой описывает, как эти сервисы
> оркестрируются в Camunda как external task workers.

## Быстрый запуск (Docker)

Требуется **Docker Desktop** (или Docker Engine + Compose).

```bash
docker compose up -d --build
```

| Сервис    | URL                       | Описание                                             |
|-----------|---------------------------|------------------------------------------------------|
| frontend  | http://localhost:8081     | SPA, прокси `/api` и `/uploads` на backend           |
| backend   | http://localhost:8080     | Go API (миграции + сидинг применяются автоматически) |
| postgres  | localhost:**5433**        | Внутри сети `postgres:5432`                          |
| pgadmin   | http://localhost:5050     | Вход: `admin@videohub.com` / `admin`                 |

Остановить с сохранением данных: `docker compose down`.
Полный сброс БД (uploads на диске сохранятся): `docker compose down -v`.

## Запуск без Docker (разработка)

```bash
# Backend
cd backend
cp .env.example .env
go run ./cmd/server

# Frontend (другой терминал)
npm install
npm run dev            # http://localhost:5173  (VITE_API_BASE_URL=http://localhost:8080)
```

## Лицензия

Учебный / дипломный проект.
